# Review Rubric Contract (content class)

**적용 대상**: `PROJECT_CLASS=content` 또는 `PROJECT_CLASS=hybrid`의 content 경로 산출물 리뷰
**목적**: reviewer가 일관된 기준으로 채점하도록 5축 채점표 제공. 설계 §3 content 매트릭스 + §5 :222 요구사항 이행.

이 파일은 **reviewer ≥ 1** 증적 규칙(설계 §3 :141-145 — `.flowset/reviews/{section}-{reviewer}.md` 파일 증거 기본)과 **evaluator `type: content`** (WI-C4 이후)의 채점 가중치 기반을 제공.

---

## 채점 5축 (요약)

| 축 | 이름 | 가중치 | 임계값 | 검증 주체 |
|----|------|-------|-------|---------|
| 1 | 사실성 (Accuracy) | 25% | ≥ 8.0 | reviewer 수동 + 출처 대조 |
| 2 | 완결성 (Completeness) | 25% | ≥ 8.0 | `completeness_checklist` 전체 done |
| 3 | 명료성 (Clarity) | 20% | ≥ 7.0 | reviewer 판정 |
| 4 | 일관성 (Consistency) | 15% | ≥ 7.0 | `style-guide.md` 준수 |
| 5 | 출처 (Source) | 15% | ≥ 8.0 | 자동 grep + 링크 작동 |

**가중치 합**: 25 + 25 + 20 + 15 + 15 = 100
**반올림 규칙**: 가중 총점은 소수 **셋째 자리에서 반올림** (`printf "%.2f"` 출력). 표시 점수와 저장 점수는 동일.
**통과 기준**: 이중 AND 게이트 — 각 축 임계값 모두 충족 **그리고** 가중 총점 ≥ 7.5.
  - 설계 의도: 한 축만 붕괴해도 REVISE(단일 축 실패로 reviewer/approver 재검토 강제) + 총점 안전망(임계값 최소 만족으로도 7.65가 나오지만, **축 3/4 임계값 미달 시 총점 7.5 미만이 가능**하여 총점 조건이 실질적으로 작동).
  - 반례 1: 축 1~5 = 8/8/7/7/8 → 가중 총점 = 7.65 → PASS
  - 반례 2: 축 1~5 = 8/8/**6**/7/8 → 축 3 임계값(7.0) 미달 → REVISE (총점 조건 이전 탈락)
  - 반례 3: 축 1~5 = 7/7/6/6/7 → 축 1/2/5 임계값(8.0) 미달 → REVISE (총점 6.75로도 탈락, 이중 게이트 동시 작동)

---

## 축 1. 사실성 (Accuracy) — 25%

내용이 출처와 대조 가능하고 사실과 일치하는가.

| 점수 | 기준 |
|------|------|
| 10.0 | 모든 주장이 출처 URL과 일치. 수치·날짜·인용 전수 검증 통과 |
| 9.0 | 1건 이하 경미한 불일치 (오탈자 수준) |
| 8.0 | 2~3건 경미한 불일치, 본문 해석에 영향 없음 |
| 7.0 | 1건 주요 불일치 발견 (해석에 영향). 재작업 필요 |
| 6.0 이하 | 2건 이상 주요 불일치 또는 출처 없이 단정 |

**감점 사유 예시**:
- 통계 수치를 출처와 다르게 인용 (축 1: -1.0)
- 인용문 원문 변형 (축 1: -1.5)
- 추측을 사실처럼 서술 (축 1: -2.0, "아마도", "~일 것이다" 표현 사용)

**검증 방법**:
- reviewer가 각 주장의 출처를 실제로 열어 확인 (최소 50% 랜덤 샘플링)
- 수치·날짜는 100% 대조

---

## 축 2. 완결성 (Completeness) — 25%

`matrix.json.sections[X].completeness_checklist`의 모든 항목이 본문에 존재·충족하는가.

| 점수 | 기준 |
|------|------|
| 10.0 | 체크리스트 전 항목 done (누락 0건) + 권장 항목 80% 이상 포함 |
| 9.0 | 전 항목 done + 권장 항목 60% 이상 |
| 8.0 | 전 항목 done, 권장 항목 선택적 |
| 7.0 | 필수 항목 중 1건 누락 — 경미 (각주 수준) |
| 6.0 이하 | 필수 항목 2건 이상 누락 또는 핵심 내용 부재 |

**필수 항목 예시** (섹션 유형별):
- 배경·목표 섹션: 문제 정의, 대상 사용자, 성공 기준
- 흐름 섹션: entry point, 각 단계, 예외 케이스, exit point
- 결론 섹션: 요약, 권고사항, 후속 조치

**검증 방법**:
- `matrix.json.sections[X].completeness_checklist` 배열 전수 대조
- 각 체크리스트 항목이 본문 H3 수준 이하에서 실제 서술되었는지 확인

---

## 축 3. 명료성 (Clarity) — 20%

독자가 **1회 읽기**로 의도를 이해할 수 있는가.

| 점수 | 기준 |
|------|------|
| 10.0 | 문장 평균 40자 이하, 모호 용어 0건, 예시 풍부 |
| 9.0 | 평균 50자 이하, 모호 1~2건 (각주로 설명됨) |
| 8.0 | 평균 60자 이하, 모호 용어 3건 이하 |
| 7.0 | 평균 70자 이하 또는 전문 용어 설명 없이 사용 |
| 6.0 이하 | 장문(80자+) 연속 또는 다중 부정/수동태 과다 |

**감점 사유 예시**:
- 다중 부정문 ("N이 아니지 않은 것은 아니다") — 즉시 -1.5
- 지시어 연쇄 ("이것의 그것의 저것") — -1.0
- 전문 용어 첫 등장 시 설명 부재 — -0.5/건
- 문장 1개에 쉼표 4개 이상 — 문장 분리 권고 -0.5/건

**검증 방법**:
- reviewer가 본문을 처음 읽을 때 **되돌아가 읽는 문장 수** 기록 (≤ 5% 목표)
- 평균 문장 길이는 `wc -w` / `.` 개수 로 근사 측정 가능

---

## 축 4. 일관성 (Consistency) — 15%

`style-guide.md` 전 규칙 준수.

| 점수 | 기준 |
|------|------|
| 10.0 | style-guide 10개 섹션 전 규칙 준수 + terminology 용어 100% 표준 |
| 9.0 | 경미한 위반 1건 이하 (예: heading 깊이 초과 1건) |
| 8.0 | 위반 2~3건, 전체 구조는 유지 |
| 7.0 | 위반 4건 또는 주요 규칙 1건 위반 (예: 코드블록 언어 누락 다수) |
| 6.0 이하 | 다수 위반 또는 파일 구조 파괴 (H5/H6 사용, 금지 용어 다수) |

**자동 강제 규칙** (reviewer 판단 전 차단):
- 코드블록 언어 미명시 → `stop-rag-check.sh` Stop hook block (수정 후 재리뷰)
- H5/H6 사용 → `stop-rag-check.sh` Stop hook block

**수동 판정 규칙** (reviewer 채점):
- 용어 통일 (표준 용어 vs 금지 표현)
- 링크·이미지 규칙 (절대 URL, alt 텍스트)
- 리스트·테이블 스타일

---

## 축 5. 출처 (Source) — 15%

모든 주장이 출처로 뒷받침되고, 출처 링크가 작동하는가.

| 점수 | 기준 |
|------|------|
| 10.0 | 섹션당 평균 출처 3건 이상 + 전 링크 200 OK + 조회 날짜 명시 |
| 9.0 | 섹션당 평균 2건 + 전 링크 작동 |
| 8.0 | **섹션당 최소 1건 (§3 기준선)** + 전 링크 작동 |
| 7.0 | 섹션당 1건이지만 링크 일부 404/리다이렉트 |
| 6.0 이하 | 섹션 중 출처 0건 존재 또는 다수 링크 사망 |

**검증 방법**:
- `stop-rag-check.sh` content 분기가 섹션별 URL 개수 자동 grep (WI-C3-content 이후)
- 링크 작동 여부는 reviewer가 수동 클릭 또는 `curl -sI -o /dev/null -w "%{http_code}"` 로 200/301 확인

**감점 사유 예시**:
- 섹션 중 1개라도 출처 0건 → 축 5 즉시 7.0 이하
- 링크 404/sinkhole → -1.0/건
- 조회 날짜 미명시 (장기 문서) → -0.5

## 증적 파일 양식

### `.flowset/reviews/{section}-{reviewer}.md` (필수)

```markdown
# Review — {section_name} by {reviewer_name}

- **리뷰 일시**: 2026-04-25T10:00:00Z
- **대상 파일**: docs/user-flow.md (commit abc1234)
- **차수**: 1 (재리뷰 시 2, 3...)

## 축별 점수
| 축 | 점수 | 근거 |
|----|------|------|
| 1. 사실성 | 9.0 | 수치 전수 대조 통과. 1건 경미 불일치(line 45, -1.0) |
| 2. 완결성 | 10.0 | completeness_checklist 5/5 done |
| 3. 명료성 | 8.5 | 평균 55자, 다중 부정 1건(line 78) |
| 4. 일관성 | 10.0 | style-guide 전 준수 |
| 5. 출처 | 9.5 | 섹션당 2건 평균, 전 링크 200 |

## 가중 총점
9.0×0.25 + 10.0×0.25 + 8.5×0.20 + 10.0×0.15 + 9.5×0.15 = 2.25 + 2.50 + 1.70 + 1.50 + 1.425 = **9.375** (반올림 규칙에 따라 소수 둘째 자리까지 표시: **9.38**)

## 판정
- [x] PASS (총점 ≥ 7.5, 전 축 임계값 충족)
- [ ] REVISE (1개 이상 축 임계값 미달)

## 수정 요청 (REVISE 시)
- line 45: "2026년 3월" → 출처 확인 결과 "2026년 4월". 수정 필요
- line 78: 이중 부정 구문 평서문으로 재작성

---
```

## 운영 규칙

| 규칙 | 적용 |
|------|------|
| 1차 판정 필수 | 파일 방식(위 양식) — `gh api` PR 리뷰는 2차 보강 (§3 :144) |
| 익명 리뷰 금지 | 파일명에 reviewer 실명 필수 |
| 재리뷰 차수 | 동일 reviewer가 여러 차례 리뷰 가능. 파일은 차수별 별도 (`-1.md`, `-2.md` 접미) 또는 단일 파일 내 섹션 분리 — 프로젝트 선택 |
| 리뷰 충돌 | 2인 이상 reviewer가 상반된 판정 시 approver가 중재. `.flowset/reviews/{section}-conflict.md`에 기록 |
| 축별 임계값 미달 | 해당 축 이유와 함께 REVISE 판정. PASS 불가 |

## 후속 WI 연계

| WI | 역할 |
|----|------|
| WI-C1 | `matrix.json.sections[X].completeness_checklist` 스키마 정의 (축 2 근거) |
| WI-C3-content | `stop-rag-check.sh` content 분기에서 축 4/5 일부 자동 감지 |
| WI-C4 | evaluator `type: content` 신설 — 본 rubric 5축을 자동 채점 축으로 차용 |
| WI-D1 | CLAUDE.md content 섹션에 본 rubric 참조 추가 |

## 변경 규칙

- 본 파일 수정 시 reviewer/approver 전원 확인 필수 (sprint 계약으로 명시)
- 축·가중치 변경은 매트릭스 스키마 영향 → 관련 hook·evaluator 동반 수정 필요
- 프로젝트별 축 확장(예: 6번째 축 "접근성") 가능. 확장 시 기존 5축 가중치 재조정
- 본 템플릿 원본은 FlowSet 저장소의 WI-B3 범위 내에서만 수정
